# gov-order — B2G 후불거래 주문서 등록 워크플로

> wcolive(애니빌드) 관리자 기준. 공공기관 전화·메일 주문을 gov1 계정으로 처리하는 표준 절차.

---

## 핵심 전제

- **gov1** = 전화/메일로 주문하는 공중보건기관 통합 wcolive 계정 (2025년부터 적용)
- wcolive의 "입금완료(발송대기)" = **실제 입금 아님**, 출고실 발송 신호일 뿐
- 실제 수금은 납품 후 서류 절차 완료 + 기관 내부 결재 후 이루어짐

---

## 주문서 등록 워크플로

### STEP 1 — 고객 정보 파악

입력 소스 우선순위:
1. 명함 이미지 → 이름 / 부서 / 전화 / 이메일 / 주소
2. 메일/문자 내용 → 품목·수량·발송 요구사항
3. 견적서 파일 (`계약서류/[기관명]/` 폴더) → 품목·단가·합계 확인
4. 이전 주문 이력 (Order.xls or Supabase) → 이전 담당자·주소 참조

### STEP 2 — CRM 기관 등록/업데이트 (Supabase)

```
기존 기관: name ilike 검색 → metadata 담당자 업데이트
신규 기관: institutions 테이블 INSERT
  - type: 공공기관(기타) | 보건소 | 전문기관
  - region, district
  - products: [알쓰패치, 노담패치 등]
  - purchase_stage: 재구매 | 구매
  - metadata.contact_name / dept / tel / email
  - metadata.required_docs: 기관별 요청 서류 목록
  - metadata.wcolive_id: gov1
  - metadata.ship_addr: 발송 주소 (상세동까지)
```

### STEP 3 — wcolive 주문서 등록

**경로:** 주문/매출관리 → 주문서관리 → 주문서 등록

#### 주문자정보
| 항목 | 입력값 |
|------|--------|
| 회원ID | `gov1` |
| 이름 | 기관명 (예: 서귀포의료원) |
| E-Mail | 기관 담당자 이메일 |
| 전화번호 | 기관 대표전화 |
| 휴대전화 | 담당자 휴대폰 |
| 요구사항 | 행사명·납기·스티커 등 특이사항 |

#### 수신자정보
| 항목 | 입력값 |
|------|--------|
| 수신자 | 담당자 이름 |
| 우편번호 | 발송지 우편번호 |
| 주소 | 발송지 상세주소 (건물+층+부서명까지) |
| 전화번호 | 기관 대표전화 |
| 휴대전화 | 담당자 휴대폰 |

#### 상품 추가 (장바구니)
- 상품 검색 → 해당 SKU 선택
- 옵션: 수량 구간에 맞는 단가 행 선택
- 주문수량: **박스 단위** (300개 = 30, 500개 = 50)
- 옵션내용: `기본리플렛, 공용스티커부채` 등 요청에 맞게 입력
- 추가 버튼 → 장바구니 확인

**주요 SKU:**
| 상품 | 상품번호 | 코드 |
|------|---------|------|
| 알쓰패치 | 2353 | APSalth02 |
| 노담패치 | 2355 | APSnodam01 |

#### 결제 및 배송정보
| 항목 | 입력값 |
|------|--------|
| 주문상태 | **입금미확인** (→ 고객 카톡+메일 자동발송 트리거) |
| 지불수단 | 계좌이체 |
| 입금은행 | 국민 778001-00-059861 주식회사 웰니스컴퍼니올리브 |
| 입금일자 | 주문 당일 |
| 입금자명 | **후불계산서** |
| 배송회사 | 로젠택배 |
| 배송비 | 0 (무료배송) |
| 처리결과 전송 | 이메일 + SMS ✅ |

### STEP 4 — 주문 등록 후 처리

1. 전체 주문서 관리에서 해당 주문 확인
2. 상세보기 → **주문상태 메모** 기록 (예: `0325 당스천` = 03/25 당일 스티커 주문)
3. Supabase CRM `current_order_no` 업데이트

### STEP 5 — 발송대기 전환 (스티커 접수완료 시)

> 스티커 접수완료 = 스티커 제작업체(성원애드피아 등)에서 주문 접수 확인받은 시점

1. 전체 주문서 관리 → 해당 주문 체크박스 선택
2. 하단 주문서처리 → "미입금 → 발송대기 상태로 변경 (입금일괄처리)" → 변경
3. 입금대기 주문서관리 → 해당 주문 체크 → **일괄입금등록** → 팝업 확인
4. 상태: 입금완료(발송대기) → 출고실 접수

### STEP 6 — 납품 후 실입금 처리 (별도)

```
납품 완료
  ↓
서류 발행: 납품서 → 청구서 → 전자세금계산서 (홈택스)
  ↓
기관 내부 결재 (기관마다 상이, 수일~수주 소요)
  ↓
실제 입금 확인 → Supabase purchase_stage 최종 업데이트
```

## 목록 없는 상품 등록 방법

> 판촉물·기념품 등 wcolive 상품목록에 없는 품목 처리

**상품번호 2374 — 인쇄비(인쇄비추가), 추가 구성물**

| 유형 | 기본가 | 용도 |
|------|--------|------|
| 1유형 | 1,000원 | 고단가 인쇄물 |
| 2유형 | 200원 | 중단가 |
| **3유형** | **100원** | **봉투류 등 저단가 — 옵션으로 단가 조합** |

**단가 조합 공식:**
```
기본가(유형 선택) + 옵션금액 = 판매단가
예) 3유형 100원 + 대봉투 옵션 220원 = 320원/부
수량 1,000 × 320 = 320,000원
```

**옵션내용 기재 예시:** `대봉투(행정봉투)` — 품목 사양 명시

## 후불 유형 구분

> `입금자명` 필드가 후불 유형 식별 키워드

| 입금자명 | 지불수단 | 실결제 방식 |
|---------|---------|-----------|
| **후불계산서** | 계좌이체 | 납품 후 세금계산서 → 계좌이체 |
| **후불카드결제** | 카드결제 | 납품 후 전화로 카드번호 수령 → Easy Check IC 앱 결제 → 전표 이메일 + 결제완료 SMS 발송 |

## wcolive 자동 알림 구조

> 별도 구현 불필요 — wcolive가 자동 처리

| 시점 | 채널 | 수신자 | 내용 |
|------|------|--------|------|
| 입금미확인 등록 | 카카오 + 이메일 | 담당자 | 주문 접수 완료 + 주문 상세 |
| 발송대기 전환 | 카카오 + 이메일 | 담당자 | 발송 준비 완료 안내 |

**주의:** 이메일 결제방법 표시가 "신용카드 결제"로 나오는 버그 있음 (실제는 후불계산서). 고객 혼동 가능성 있으나 wcolive 시스템 한계.

**자동화 범위:** 주문서 등록까지만. 알림 발송은 wcolive가 담당 → 중복 발송 금지.

## 자동화 목표 (FLUX X5)

| 단계 | 현재 | 목표 |
|------|------|------|
| 고객 정보 파악 | 메일/명함 수동 확인 | 메일·문자 자동 파싱 |
| CRM 등록 | 수동 Supabase INSERT | 자동 매칭·등록 |
| wcolive 주문서 | 수동 입력 | 초안 자동 생성 → 대표 승인 |
| 상태 변경 | 수동 클릭 | 트리거 기반 자동 전환 |
| 서류 발행 | generate_invoice.py 반수동 | 주문번호 입력 → 전체 자동 |
