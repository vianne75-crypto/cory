# KISS 2026 백엔드 — 대표 실행 가이드

> 이 문서 하나만 위에서 아래로 따라가면 됩니다.
> **먼저 안심**: KISS 신청·카운터·진단은 **지금 라이브에서 정상 작동**합니다(신청 저장·카운터·진단 응답 전부 실측 통과). 아래 3개는 "안 되는 것 고치기"가 아니라 **더 안전하게 만들기**입니다. **오늘 인쇄(7/5)를 막는 항목은 없습니다.**

---

## 0. 지금 상태 한눈에

```
[학회 참석자] --QR스캔--> aps7.net/kiss2026 (랜딩, PIXEL)
                              │
                              │ 신청/카운터/진단
                              ▼
                    aps-lead.vianne75.workers.dev  ← FLUX Worker (라이브·정상)
                              │
                              ▼
                    Supabase (cory DB)
                     ├ kiss_signups   (신청자)     ✅ 작동
                     ├ kiss_diagnosis (진단 6문항)  ✅ 작동
                     └ kiss_flywheel  (구매의향)    ⚠️ 아직 유실 중 (워크스트림 3)
                              │
                              ▼
                    cory 관리자 → "KISS 리드" 탭에서 확인 ✅
```

**용어 3개만 알면 됩니다:**
- **Worker** = 신청을 받아 DB에 저장하는 중간 서버 (Cloudflare). 이름 = `aps-lead`.
- **Supabase** = 데이터가 쌓이는 데이터베이스.
- **ref_key** = 전화번호를 암호화(해시)한 값. 설문과 연락처를 "전화번호 노출 없이" 연결하는 열쇠.

## 워크스트림 1 — 법적 게이트 (대폭 단순화됨 · 대표 결정 2026-07-05)

### 결정: 민감정보 비수집
대표 결정으로 **민감정보를 아예 안 모으기로** 했습니다. 실제 확인 결과 파이프라인의 유일한 민감 항목은 `quiz_answer`(폭음 지식 퀴즈 답)뿐이었고, 본인 체질 결과는 원래 저장 안 함.

→ **FLUX 반영 완료(7/5)**: Worker가 `quiz_answer`를 프론트가 보내도 **저장하지 않음**. 파이프라인에 민감정보 **0건**.
→ 결과: BOND 개인정보보호법 §23(민감정보 별도 동의) 요건이 **"비수집"으로 자동 충족** (가장 강력한 준수).

### 남은 것 (급하지 않음 · 위생 정리)
| # | 할 일 | 누가 | 급한가 |
|---|------|------|--------|
| 1-A | 프론트 **민감정보 동의 체크박스 제거** + privacy §3-2 삭제/단순화 | PIXEL | 학회 전 권장 (수집 안 하는데 동의 문구 있으면 부정확) |
| 1-B | BOND에 **비수집 확정 통보** → 법적 기록 갱신 | FLUX→BOND | 🔴 7/6 게이트 (통보만) |
| 1-C | `ref_phone` 빈 컬럼 삭제 (선택·정리) | 대표 SQL | 여유 시 |
| 1-D | 전용 salt 등록 (선택·이제 법적 아님) | 대표 | 여유 시 |

### 어떻게

**1-C. (선택) ref_phone 컬럼 삭제** — Supabase SQL Editor:
```sql
UPDATE public.kiss_diagnosis SET ref_phone = NULL WHERE ref_phone IS NOT NULL;
ALTER TABLE public.kiss_diagnosis DROP COLUMN IF EXISTS ref_phone;
```

**1-D. (선택) 전용 salt** — 터미널:
```bash
cd /Users/olive/Qsync/cory/workers && wrangler secret put PRIVACY_HASH_SALT
```

> ⚠️ 이제 1-C·1-D는 **법적 필수 아님** (민감정보 자체를 안 모으므로). 순수 위생 정리.
> **진짜 필수는 1-B** (BOND에 "이제 민감정보 안 모읍니다" 통보) + 1-A (프론트 동의 UI 정리).

## 워크스트림 2 — 학회 봇 방어 (마감 🔴 학회 7/6~10)

### 왜 하나
신청 폼이 **누구나 열 수 있는 공개 주소**입니다. 나쁜 프로그램(봇)이 가짜 신청 100건을 순식간에 넣으면:
- 카운터가 "선착순 100명 마감"으로 바뀜 → **현장의 진짜 보건관리자가 신청 못 함**
- 가짜 이름·주소가 쌓여 실물 체험팩이 엉뚱한 곳으로 발송

우리가 이미 넣은 방어: 같은 전화 24시간 1건, 같은 IP 5분 1건.
**부족한 것**: 봇이 IP를 바꿔가며 하면 뚫림 → 근본 차단은 아래.

### 무엇을 (2가지)
| # | 할 일 | 누가 | 효과 |
|---|------|------|------|
| 2-A | **Cloudflare Turnstile** (사람 확인 위젯) | PIXEL(프론트) + 대표(CF 설정) | 봇 대량신청 근본 차단 |
| 2-B | **당일 운영 런북** (조기 마감·리드 팔로업 절차) | FLUX 작성 | 현장 대응 |

### 어떻게

**2-A. Turnstile** — cloudflare의 무료 "로봇 아님" 확인 (구글 reCAPTCHA 같은 것).
- PIXEL에게 발주 필요 (핸드오프는 아래 §참고에 준비됨).
- 대표는 Cloudflare 대시보드에서 Turnstile 사이트키 발급만 하면 됨 (5분).
- **판단**: 학회가 오프라인 현장이라 봇 유입 확률은 낮음. 위험 대비 여유 있으면 학회 첫날 관찰 후 결정해도 됨. **필수는 아니고 보험**.

**2-B. 운영 런북** — FLUX가 작성. 내용:
- 조기 마감법 (재고 소진 시): 아래 한 줄 SQL
  ```sql
  UPDATE public.settings SET value = jsonb_set(value,'{closed}','true') WHERE key='kiss_counter';
  ```
  (다시 열기는 `'false'`)
- 리드 확인: cory 관리자 → KISS 리드 탭 → 새로고침
- 구매의향(direct) 리드 우선 팔로업 → farm360 이관

## 워크스트림 3 — Fast-follow (마감 무관, 여유 있을 때)

### 왜 하나
두 가지 개선. 지금 당장 안 해도 학회는 돌아감.

### 무엇을 (2가지)
| # | 할 일 | 문제 | 파일 |
|---|------|------|------|
| 3-A | **원자적 카운터** | 동시 신청이 겹치면 100명을 몇 건 초과 저장될 수 있음 | `data/kiss-atomic-counter.sql` |
| 3-B | **플라이휠 유실 복구** | 완료화면 "직접구매/공공/문의" 버튼 선택이 **지금 저장 안 됨**(레거시 프록시가 권한 없이 저장 시도→차단) | `data/kiss-flywheel-schema.sql` |

### 어떻게

**3-A. 원자적 카운터** (초과신청 방지):
1. `data/kiss-atomic-counter.sql` → Supabase 실행 (함수 생성)
2. FLUX가 Worker를 이 함수 호출로 전환 (코드 작업)
→ 무료 체험팩이라 초과 몇 건은 큰 문제 아님. 여유 있을 때.

**3-B. 플라이휠 복구** (구매의향 신호 저장):
1. `data/kiss-flywheel-schema.sql` → Supabase 실행 (테이블 생성)
2. FLUX가 Worker에 `/kiss-flywheel` 엔드포인트 추가
3. PIXEL이 프론트 버튼을 새 엔드포인트로 전환
→ **구매의향(direct)은 가장 뜨거운 영업 리드**라 학회 후 팔로업 가치 큼. 우선순위 높게 권장.

## 🎯 학회 전(오늘 7/5) 대표가 실제로 할 최소 목록

인쇄는 그냥 진행하세요(백엔드 정상). 진짜 필수는 아래 **①만** — 카운터가 테스트로 오염돼 "신청 N명"이 가짜로 뜹니다.

**① 🔴 필수 — 테스트 데이터 정리 + 카운터 0부터 시작** (Supabase SQL Editor):
```sql
-- 신청 테스트 행 전체 삭제 + 카운터 리셋 (내일 진짜 0명부터)
TRUNCATE public.kiss_signups RESTART IDENTITY CASCADE;
-- 진단 테스트 행 정리
DELETE FROM public.kiss_diagnosis WHERE utm_source IN ('diag_test','harden_test');
-- 확인
SELECT * FROM public.kiss_counter_view;   -- current_count 0 이어야 함
```

**② 권장 — BOND에 비수집 통보** (FLUX가 INBOX로 처리 가능)

**③ 선택 — ref_phone 삭제·salt** (§워크스트림1 1-C/1-D, 법적 필수 아님)

나머지(Turnstile·런북·원자카운터·플라이휠)는 학회 직전~직후 순차 처리해도 안전.
